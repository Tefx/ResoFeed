## Authority and Boundary Lock

Remediation MUST preserve these constraints:

- `docs/DESIGN.md` and `docs/ui-preview.html` are the UI/runtime visual authorities unless a row below records a true spec conflict. No such conflict is currently recorded.
- `docs/ARCHITECTURE.md` hard boundaries remain in force: one `resofeed serve` Go process, one SQLite+FTS5 store, lexical retrieval only, thin HTTP/MCP transports, no accounts/OAuth/RBAC, no service/repository/DI/event-bus layer, no sync/merge coordinator, no durable jobs/queues/activity ledgers for ingest, and no settings dashboard.
- `AGENTS.md` project invariants remain in force where available to agents: no vector DB/RAG, no accounts/OAuth/per-agent registry, no settings dashboard, no Java-style app/domain/service/repository layers, no forbidden UI concepts such as folders/tags/unread/mark-all-read/settings sliders/drag ordering.
- Acceptance criteria below are limited to UI/runtime conformance proof. Product fixes must happen in downstream implementation steps.

## Proof Path Legend

- `expected-red test`: a failing automated test or component/unit assertion that captures the current divergence before remediation.
- `runtime/liveness proof`: a live app/browser/API proof that a runtime route, command, or state transition works.
- `browser artifact`: DOM snapshot, accessibility snapshot, computed style capture, or screenshot from the browser.
- `UIUX screenshot audit`: visual comparison against `docs/ui-preview.html` and tokenized `docs/DESIGN.md` expectations.
- `non-red justification`: explicit reason a pre-remediation red test is not meaningful or is blocked, plus downstream proof that closes it.
