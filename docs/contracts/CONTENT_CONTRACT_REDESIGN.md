# ResoFeed Content Contract Redesign

## Status

Proposed architecture contract.

## Original Request

Design a complete, clean, and reasonable system solution for ResoFeed's content processing problems:

- Some titles are localized to Chinese while others remain English or degrade into URLs.
- One-time prompts such as “核心洞察要分点” can produce decode errors.
- The previous `rss-agent` Key Points design is not reflected in the current ResoFeed mainline.
- Re-ingest failures must not overwrite existing usable titles, summaries, insights, or previous content.
- Feed rows must remain compact and must not show Key Points.
- Inspector output and generated reading content should be fully Chinese-localized.
- Historical items should be re-ingested after the new contract is introduced.
- Key Points must be fixed at 3–5 items.

## Executive Summary

ResoFeed should treat generated item content as a structured content contract, not as a small set of overloaded text fields.

The clean target design is:

1. Split source titles from localized display titles.
2. Add first-class `key_points` as a structured array with 3–5 Chinese, source-grounded items.
3. Keep `core_insight` as one concise Chinese sentence, but route all “分点 / 列表 / 要点” requests into `key_points`.
4. Separate persisted content status from the last reprocess attempt status.
5. Scope one-time prompts and Steer rules to content emphasis and field selection, never schema mutation.
6. Render Key Points only in Inspector, never in Feed rows.
7. Run a full historical re-ingest after schema, prompt, validation, transport, and UI contracts are upgraded.

The product goal is not visual compression for its own sake. The goal is fast, low-friction acquisition of high-density information. Structured lists, clear grouping, and stable field semantics support that goal better than paragraph-only compression.

## Non-Goals

- Do not introduce vector databases, embeddings, RAG, semantic search, or external retrieval substrates.
- Do not add accounts, roles, multi-user auth, agent registries, sidecars, workers, or extra deployable services.
- Do not allow arbitrary Markdown or HTML rendering from model output.
- Do not show Key Points in Feed rows.
- Do not let user prompts or source text alter JSON schema, provenance rules, status values, or persistence semantics.

## Constraints Preserved

- One deployable Go binary remains the only runtime unit.
- SQLite plus FTS5 remains the only storage and search substrate.
- OpenRouter / LLM remains a bounded JSON-in / JSON-out transformer.
- HTTP and MCP expose the same product concepts.
- Single owner token remains the delegation boundary.
- `actor_id` remains provenance/idempotency, not authorization.

## Current Problems

### Problem 1: Title Localization Is Ambiguous

The current content contract overloads `title`.

Sometimes the model treats it as a generated user-readable field and localizes it to Chinese. Other times it treats it as source/provenance and preserves English. In worse cases, fallback paths can display a URL-like title.

Root cause: the schema does not distinguish source title truth from localized display title.

### Problem 2: One-Time Prompt Formatting Requests Can Conflict With Schema

Requests such as “核心洞察要分点” currently collide with the invariant that `core_insight` must be a single sentence. If the model follows the user and emits multiple sentences or list formatting in `core_insight`, strict validation can classify the output as a decode error.

Root cause: the contract has no legal field for structured Key Points, so user intent is forced into the wrong field.

### Problem 3: Key Points From the Older RSS Agent Design Were Not Migrated

The older `rss-agent` design had explicit `Key Points`. The current ResoFeed schema has `summary` and `core_insight`, but no structured `key_points` field.

Root cause: the current schema collapsed reading structure into plain text fields.

### Problem 4: Re-Ingest Failure Can Degrade Usability

Re-ingest failures such as decode errors should be an attempt-level failure. They must not erase the currently usable item content.

Root cause: current status and persistence semantics conflate “current content state” with “latest processing attempt state”.

## Target Content Model

### Logical Fields

```text
source_item_title              // original article title from RSS/source, literal provenance
localized_title                // Chinese display title generated by model
summary                        // Chinese contextual summary
core_insight                   // exactly one concise Chinese sentence
key_points                     // structured Chinese array, fixed 3–5 items
value_tier                     // stable product label: high / brief / source-claim
content_status                 // status of currently persisted generated content

last_reprocess_status          // status of the latest reprocess attempt
last_reprocess_error_code      // decode_error / provider_error / invalid_model / timeout-like status
last_reprocess_error_message   // user-safe localized explanation
last_reprocess_at              // timestamp of latest attempt
```

### Field Semantics

#### `source_item_title`

Owner: source ingestion / RSS extraction.

This field preserves the original article title as evidence. It should not be localized or rewritten by the LLM.

#### `localized_title`

Owner: model generation after validation.

This field is the display title when the processing language is Chinese. It must be Chinese unless no model-backed content is available.

#### `summary`

Owner: model generation after validation.

This field provides Chinese context and explanation. It may contain compact paragraph structure, but it is not the primary container for lists.

#### `core_insight`

Owner: model generation after validation.

This field is exactly one concise Chinese sentence that answers “why this matters”. It is not a list and should not contain multiple bullets.

#### `key_points`

Owner: model generation after validation.

This field is a structured array of 3–5 Chinese items. Each item must be source-grounded, independently useful, and high-density.

Key Points are required for every successful generated output.

#### `content_status`

Owner: backend validation and persistence.

This status describes the currently persisted generated content. It should not be overwritten by a failed reprocess attempt.

#### `last_reprocess_*`

Owner: reprocess runtime.

These fields describe the latest attempt. They can record failure without degrading existing content.

## JSON Output Contract

The model-facing successful output should be equivalent to:

```json
{
  "localized_title": "中文标题",
  "summary": "中文上下文摘要。",
  "core_insight": "一句话中文核心判断。",
  "key_points": [
    "中文高密度要点一。",
    "中文高密度要点二。",
    "中文高密度要点三。"
  ],
  "value_tier": "high",
  "model_status": "ok"
}
```

### `key_points` Rules

- `minItems`: 3
- `maxItems`: 5
- Language: Chinese when target language is Chinese.
- Must be source-grounded.
- Must not be generic filler.
- Must not duplicate `core_insight` verbatim.
- Must be suitable for Inspector list rendering.

Valid examples:

- “Manus 因中国监管指令撤销 Meta 收购案，说明跨境 AI 并购已经受到实质监管约束。”
- “该事件会影响 AI 初创公司的退出预期，因为买方意愿不再等同于交易确定性。”
- “对读者的价值在于判断 AI 公司估值时必须同时考虑技术、资本和监管三条线。”

Invalid examples:

- “值得关注。”
- “影响重大。”
- “这篇文章讨论了相关问题。”
- Empty or near-empty list items.

## Prompting Hierarchy

Prompt inputs should be compiled into explicit priority layers.

```text
System/schema contract
  > source grounding and provenance rules
  > display/content field contract
  > active Steer rules
  > one-time prompt
  > source text as evidence only
```

### System Contract

System contract owns:

- JSON schema.
- Required fields.
- Chinese localization rules.
- Source grounding.
- Literal provenance preservation.
- `core_insight` single-sentence invariant.
- `key_points` 3–5 item invariant.
- Stable status and enum values.

System contract cannot be overridden by Steer, one-time prompt, source text, feed text, source titles, URLs, or item metadata.

### Steer Rules

Steer rules are durable preference and ranking guidance.

Allowed effects:

- Influence fact selection.
- Influence value tier judgment when source-backed.
- Influence which points are emphasized in `summary` and `key_points`.
- Influence ordering of Key Points.

Forbidden effects:

- Alter schema.
- Alter status values.
- Override target language.
- Override source grounding.
- Turn `core_insight` into a list.

### One-Time Prompt

One-time prompt is current-item guidance.

Allowed effects:

- Influence the emphasis of `summary`.
- Influence the focus and wording of `key_points`.
- Influence the angle of `core_insight` while preserving single-sentence shape.

Forbidden effects:

- Alter schema.
- Emit Markdown headers or code fences.
- Set status values.
- Suppress required fields.
- Make `core_insight` multi-sentence or list-shaped.

### Intent Routing Examples

User one-time prompt:

```text
核心洞察要分点
```

Compiled interpretation:

```text
Keep core_insight as exactly one Chinese sentence.
Place multi-point insight content in key_points.
Use summary for supporting context.
```

User one-time prompt:

```text
请列出实现风险
```

Compiled interpretation:

```text
Focus key_points on source-backed implementation risks.
Keep summary contextual.
Keep core_insight one sentence.
```

## Localization Rules

### Fully Chinese-Localized Generated Content

When processing language is Chinese, the following must be Chinese:

- `localized_title`
- `summary`
- `core_insight`
- `key_points`
- user-facing error/status text
- Inspector labels such as “摘要”, “核心洞察”, “要点”, “本文重处理”

### Literal Provenance Must Remain Literal

The following remain unchanged:

- URLs
- source IDs
- source URLs
- source titles such as “TLDR AI Feed”
- product/company names unless there is a conventional Chinese rendering
- exact quoted terms from the source

### Title Display Rule

Display title should resolve as:

```text
localized_title if valid
else source_item_title if valid
else safe fallback title
```

URL-like strings must not replace a valid existing title after failed reprocess.

## Re-Ingest Semantics

### Successful Re-Ingest

On successful validation, atomically replace current generated content:

```text
localized_title
summary
core_insight
key_points
value_tier
content_status
```

Then update:

```text
last_reprocess_status = ok
last_reprocess_error_code = null
last_reprocess_error_message = null
last_reprocess_at = now
```

### Failed Re-Ingest

On failed validation, timeout, invalid model, provider error, decode error, or unavailable result:

Do not update:

```text
localized_title
source_item_title
summary
core_insight
key_points
value_tier
content_status
```

Only update:

```text
last_reprocess_status = failed
last_reprocess_error_code = <safe code>
last_reprocess_error_message = <localized safe message>
last_reprocess_at = now
```

The UI should continue showing the existing content and show a non-destructive status line:

```text
上次重处理失败 · 解码错误 · 已保留现有摘要
```

## UI Contract

### Feed Row
Feed rows must not show Key Points.

Feed row display:

```text
TLDR AI FEED · 1m · 全文 · 模型支持 · 高价值
中文本地化标题
一句话核心洞察预览……
```

Rationale: Feed is the scanning surface. The title says what the item is about; the preview should prioritize the validated `core_insight` so the row answers why the item may deserve inspection or resonance. Fall back to `summary` only when `core_insight` is missing, and to source-backed excerpt only when generated preview is unavailable. Adding Key Points, duplicate Summary + Core Insight blocks, or repeated visual prefixes to Feed rows would increase row height and reduce scan speed.
### Inspector
Inspector is the structured reading surface.

Inspector display:

```text
INSPECTOR

中文本地化标题

ORIGINAL  TLDR AI Feed
LINKS     原文链接 · 来源链接
AI STATUS 模型支持 · 全文 · 质量：高价值
ATTEMPT   失败 · 解码错误 · 已保留现有摘要和要点

摘要：
中文上下文摘要。

核心洞察：
一句话中文核心判断。

要点：
• 中文高密度要点一。
• 中文高密度要点二。
• 中文高密度要点三。

本文重处理
[重新处理本文]
```

### Key Points Rendering

Frontend should render Key Points from structured data, not from raw Markdown.

Recommended HTML semantics:

```text
section aria-label="要点"
  p label: 要点：
  ul
    li ...
```

The model should not emit HTML.

## Historical Re-Ingest Plan

After schema, prompt, validation, API/MCP transport, and UI changes are in place:

1. Run a compatibility migration.
2. Seed old rows:
   - `source_item_title = current title`
   - `localized_title = current title`
   - `key_points = []`
   - `content_status = existing model_status`
   - `last_reprocess_* = null`
3. Mark eligible existing items for content-contract re-ingest.
4. Re-ingest using the new content contract.
5. On success, replace generated fields atomically.
6. On failure, preserve existing content and record only last attempt state.

### Historical Data Limitation

For historical rows whose original source title was already overwritten, the system may not be able to perfectly reconstruct source provenance from the existing database alone. Re-ingest can repair localized display fields, but exact original titles depend on retained RSS/source data or re-fetch availability.

## Validation Contract

Validation must reject successful model outputs that violate:

- missing required field;
- non-Chinese generated fields when target language is Chinese;
- `core_insight` is empty, multi-sentence, or list-like;
- `key_points` count is not 3–5;
- any `key_points` item is empty, generic filler, or not source-grounded;
- invalid `value_tier`;
- invalid `model_status`;
- output includes extra fields;
- output leaks prompt-injection instructions.

Validation failure must become a non-destructive attempt failure, not a destructive content update.

## FTS Contract

SQLite FTS should index current readable content:

- `localized_title`
- `source_item_title`
- `summary`
- `core_insight`
- joined `key_points`
- source/provenance URLs and identifiers
- `value_tier`

Failed re-ingest must not pollute FTS with failed candidate content. If existing content is preserved, FTS should remain aligned with preserved content.

## Architecture Decisions

### ADR-1: Key Points Are a First-Class Structured Field

Decision: Add structured `key_points` with 3–5 items.

Rationale: Lists are a legitimate high-density reading format. They reduce cognitive load and give user one-time prompts a valid output slot.

Trade-off: Requires schema, validation, persistence, API/MCP, frontend, tests, and historical re-ingest changes.

Fails if: The UI later tries to render raw Markdown instead of structured arrays.

### ADR-2: Feed Does Not Show Key Points

Decision: Key Points are Inspector-only.

Rationale: Feed is a scanning surface. Inspector is the structured reading surface.

Trade-off: Users must open Inspector for full structured detail.

Fails if: Feed cards later expand into mini-articles and lose scan speed.

### ADR-3: Source Title and Localized Title Are Separate

Decision: Split `source_item_title` and `localized_title`.

Rationale: Eliminates ambiguity between literal provenance and Chinese display output.

Trade-off: Historical data needs compatibility seeding and re-ingest.

Fails if: Future prompt/schema reintroduces a single overloaded `title` field.

### ADR-4: Re-Ingest Failure Is Attempt State, Not Content State

Decision: Failed attempts update only `last_reprocess_*` fields.

Rationale: Existing usable content should remain usable after failed processing.

Trade-off: Status model becomes more explicit and has more fields.

Fails if: UI uses last attempt failure to hide current content.

### ADR-5: One-Time Prompts and Steer Rules Are Field-Scoped Guidance

Decision: One-time prompts and Steer rules can influence emphasis and fact selection but cannot alter schema or field invariants.

Rationale: User intent should be honored inside safe slots, not treated as schema authority.

Trade-off: Some user phrasing is interpreted rather than followed literally.

Fails if: Prompt compilation passes raw user intent as higher-priority instruction.

## Architecture Basis

```yaml
architecture_basis:
  system_layers:
    - layer: core_content_contract
      responsibility: Defines generated item fields, invariants, validation, and status semantics.
    - layer: prompt_compilation
      responsibility: Compiles system contract, Steer rules, one-time prompt, and source text into bounded LLM payload.
    - layer: llm_adapter
      responsibility: Sends JSON schema request and returns candidate output without owning durable state.
    - layer: persistence
      responsibility: Atomically commits successful content and records non-destructive attempt failures.
    - layer: transport
      responsibility: Exposes identical HTTP and MCP item/re-ingest contracts.
    - layer: frontend
      responsibility: Renders Feed compactly and Inspector structurally in Chinese.

  source_of_truth_matrix:
    source_item_title: RSS/source extraction
    localized_title: validated model output
    summary: validated model output
    core_insight: validated model output with single-sentence invariant
    key_points: validated model output with 3-5 item invariant
    content_status: backend validation state of currently persisted content
    last_reprocess_status: latest reprocess attempt state
    source_urls_and_identifiers: source/provenance extraction, never localized

  internal_module_map:
    prompt_contract:
      owner: internal/resofeed

      owner: web/src/routes/components/Feed.svelte

      - last_reprocess_status

      - explicit request context
      - existing single-binary reprocess flow
      - SQLite transactions for commit boundaries

      - direct module calls inside existing Go binary
      - no DI container or event bus

  shared_abstractions:
    shared_types:

      - name: ReprocessAttemptState

          - reprocess module/direct functions

    shared_utilities:
      - name: StructuredContentValidation

      owner: internal/resofeed

  ux_surfaces:
    - surface: Feed list

    - surface: API/server
      launch_or_entrypoint: cmd/resofeed
      minimum_liveness_proof: HTTP/MCP response proof exposing localized title, key_points, content status, and last reprocess attempt state
```

## Implementation Handoff

### Design Scope

This document defines the target content contract for Chinese localized generated content, structured Key Points, field-scoped prompt guidance, non-destructive re-ingest failures, and historical re-ingest.

### Suggested Implementation Order

1. Add backend content/attempt-state contract types and validation rules.
   - Reason: All later layers depend on the canonical contract.
2. Add SQLite migration and compatibility seeding.
   - Reason: Persistence must understand new fields before API/UI rely on them.
3. Update prompt schema and prompt compilation.
   - Reason: Model needs legal slots for localized title and Key Points.
4. Update reprocess persistence semantics.
   - Reason: Failure must preserve current content before historical re-ingest runs.
5. Update HTTP/MCP transport contracts.
   - Reason: Frontend and agents need the same fields.
6. Update frontend API types and rendering.
   - Reason: Feed and Inspector behavior diverge intentionally.
7. Add tests for title localization, Key Points, prompt routing, Steer fusion, re-ingest failure preservation, and FTS alignment.
8. Run full historical re-ingest.
   - Reason: Existing content needs the new localized and structured fields.

### Watch For

- Do not make Key Points a Markdown string.
- Do not show Key Points in Feed rows.
- Do not let failed re-ingest hide or degrade current content.
- Do not collapse `content_status` and `last_reprocess_status` back into one field.
- Do not let one-time prompts or Steer rules set model status or alter schema.
- Do not translate source identifiers, URLs, or source names.

## Verification Matrix

| Requirement | Required Proof |
| --- | --- |
| Feed excludes Key Points | Browser/DOM proof for Feed item with generated key_points |
| Inspector shows Key Points | Browser/DOM/accessibility proof showing 3–5 Chinese list items |
| Title localization stable | Backend/API test proving source title literal and localized title Chinese |
| One-time prompt “核心洞察要分点” succeeds | Backend test proving `core_insight` one sentence and `key_points` populated |
| Steer + one-time prompt fusion | Test proving Steer affects fact selection without schema drift |
| Re-ingest failure preserves content | Persistence/API/UI test proving old title/summary/core/key_points remain visible |
| FTS indexes Key Points | Search test over a Key Point phrase |
| Historical re-ingest completes safely | Runtime report with attempted/updated/failed counts and non-destructive failures |

## Open Questions

- Historical title fidelity depends on whether original RSS/source title data remains available.
- The exact migration field names should be chosen to match existing Go/JSON naming conventions.

## Certainty

- `[Proven]` Current mainline lacks first-class Key Points and treats summary/core insight as plain text fields.
- `[Likely]` First-class structured Key Points plus separated content/attempt state is the cleanest long-term solution.
- `[Speculative]` Historical title recovery quality depends on retained source data and feed availability.
